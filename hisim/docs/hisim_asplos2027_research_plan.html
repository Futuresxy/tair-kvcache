Decision first

0. 先确定论文最小闭环

这项工作可以长成一个很大的平台，但 ASPLOS 2027 秋季轮只剩约 59 天。投稿版本必须围绕一个可证伪、可复现的系统命题收敛；“多支持几种硬件和策略”本身不是足够强的研究贡献。

推荐论文主线：构建一个保留真实 serving framework 控制流、又能抽象多种加速器和多级 KVCache 的 framework-in-the-loop 仿真器；在此基础上提出 benefit-per-byte / reuse-deadline 感知的分层放置与预取策略，证明它能在不同硬件、负载复用度和存储带宽下稳定改善 SLO attainment 或 goodput，而不是仅提高命中率。

投稿必须完成

  • H100 或 H20 上的仿真—实机闭环校准。
  • 第二种差异明显的目标平台：优先 RTX 4090；若已有 910C 机器和权限则换成 910C。
  • SGLang 后端端到端可复现；vLLM 至少完成一条验证链路。
  • LRU/FIFO/LFU + 一个新策略，HBM/DRAM/SSD 三层。
  • 误差、仿真速度、策略收益、消融、失败案例。

可延后到投稿后

  • 同时完整支持 910C、4090、H20、H100 的全部模型组合。
  • 分布式多节点、网络拥塞、PD 分离的全细节。
  • 大量启发式策略和自动搜索器。
  • 真实 SSD 数据面与所有 KVCM storage backend。
  • 统一 SGLang/vLLM 每一个版本的兼容层。

2026-08-09 Go / No-Go

若届时没有“至少两种硬件、两类 workload、端到端延迟 MAPE 可接受”的验证闭环，应放弃在 9 月稿件中宣称跨加速器通用性，改投更聚焦的 cache-policy study，继续把平台打磨到下一周期。

现有本地实验能说明什么

ShareGPT 300

cold prefix reuse 约 4.7%；warm storage prefetch 约 29%。这表明 HiCache 路径被触发，但 warm TPOT/E2E 未必更好，说明“更高命中率”不等于“更好系统性能”。

SWE-bench agentic trace

40 请求 cold/warm prefix reuse 约 91.4%/94.9%；300 请求 warm 的 prefix reuse 反而下降，但延迟明显改善。它再次证明必须联合观察队列、L2 load、prefetch 和 E2E。

这些数值来自当前工作区已生成报告，只能作为功能性 baseline，不能替代与真实硬件对齐的精度实验；现有 H100 结果依赖 predictor 数据和配置假设。

必须先修的正确性问题：① MockHiCacheStorage.init_kvcm() 当前把所有请求写入固定的 instance_id，多实例实验可能产生跨 Instance 复用假象；正确 key 至少应包含 instance_id + model/KV schema fingerprint + prefix block hash。② ConfigManager.get_kv_cache_bytes() 及 per-layer 版本当前乘的是模型 data_type.bytes，而不是 kv_cache_data_type.bytes；例如 FP8 权重 + FP16 KV 会低估一半迁移字节数。修复并重跑验证前，现有 L2 迁移时长只能视为功能性结果。

必须编码成断言的系统不变量

Namespace

match(a,b) ⇒ a.instance_id == b.instance_id；即使 token、模型名、hash 完全相同也不可跨 Instance 复用。

Schema

同一 Instance 内模型/KV dtype、block size、TP/PP、layout 必须固定；变化时建立新 instance 或新 schema generation。

Residency

只有 serving 状态的完整 LocationSpec 才能命中；writing/deleting 不可被读取。

Capacity

每次事件后各 tier resident bytes 不超过容量；同时统计元数据与碎片开销。

Causality

事件完成时间不早于依赖完成时间；同一资源上的冲突事件必须串行或按明确定义共享带宽。

Prefix

命中必须是从根开始的连续合法前缀；不能把不连续块数直接等价成可跳过的 prefill tokens。

Cross-accelerator modeling

5. H100/H20、RTX 4090、Ascend 910C 如何建模

不把峰值 TFLOPS/带宽直接当预测器。每个平台都走同一套“静态事实 → 微基准 → iteration profile → 校准 → 独立验证”的 onboarding protocol。厂商未公开或不可信的 910C 参数，用实测黑盒 profile 代替猜值。

平台仓库当前状态投稿可用条件主要风险
H20外部 AIC 数据包 + 配置补充真实 serving validation，冻结驱动/框架/模型版本。现有 scale factor 是否对未见 workload 仍成立。
H100 SXM场景配置 + AIC system作为最先闭环的 reference platform。配置中的 8 GB/s storage、64 GB/s host bandwidth 是场景假设，不是 H100 固有属性。
RTX 4090本地注册表已有 24 GB / 1008 GB/s新建 profile bundle、场景 config、实机 validation。无 NVLink、24 GB 容量、GeForce 软件栈与数据中心 GPU 差异显著；理论 FLOPS 容易误导。
Ascend 910C未注册；仅有本地 910B/950 条目以真实机器采集 CANN/框架 iteration profile；注册数据注明来源与置信度。公开可核验规格有限；AIConfigurator 后端未必直接支持；算子、通信和内存语义与 CUDA 不同。

微基准矩阵

计算与通信

  • prefill：batch size × total prompt tokens × prefix length × chunk size。
  • decode：batch size × context length 分布，不只记录平均值。
  • TP/PP/EP：all-reduce/all-to-all 的消息大小与 topology。
  • 模型：至少一个 dense（Qwen3-8B）与一个较大/稀疏模型。
  • dtype：模型权重 dtype 与 KV dtype 分开扫描。

数据移动

  • H2D/D2H：4 KB 到数 GB、连续/分段、pinned/pageable、并发流。
  • DRAM：单/双路 NUMA、socket-local/remote、内存压力。
  • SSD：块大小、queue depth、direct/buffered I/O、冷/热 page cache。
  • 重叠：compute + H2D、compute + D2H、read + write 同时发生。
  • 尾延迟：P50/P95/P99，不能只用最大带宽。

模型结构：先可解释，再学 residual

Tevent = Tanalytical/profile lookup(features) × scale + fresidual(features),  并为外插点输出 uncertainty / out-of-domain 标志。
  • 基础层：AIConfigurator、查表插值或 roofline-like 可解释模型。
  • 残差层：只拟合系统误差，不让 XGBoost 隐式吞掉所有物理语义。
  • 训练/测试：按 workload/model/config 分组切分，禁止同一 trace 的相邻 batch 同时出现在训练和测试。
  • 报告：iteration latency MAPE、端到端 TTFT/TPOT MAPE、P95 absolute percentage error、Spearman 排序相关性。
910C 的建议路径：若 2026-07-26 前无法获得稳定机器、CANN 版本和 profiling 权限，9 月稿件不要把 910C 放进主结论；可以把它作为“新后端接入方法”或 appendix case study。用 910B 的理论规格代替 910C 会直接伤害可信度。

Framework integration

6. vLLM 接入：复用 Connector，但不要混淆两件事

仓库已有 kv_cache_manager/py_connector/vllm，实现了 vLLM V1 的 KVCM 数据传输与 location 查询。这说明真实数据面已经有接点；但 HiSim 的仿真层仍只有 SGLang。接入 vLLM 需要新增仿真 adapter，不能只把 benchmark 的 --backend 改成 vLLM。

已有真实 Connector

  • TairKvCacheConnector 继承 vLLM KVConnectorBase_V1。
  • get_num_new_matched_tokens() 查询外部前缀。
  • register_kv_caches() 获取 paged KV buffers。
  • start_load_kv()、save_kv_layer()、request_finished() 处理数据生命周期。
  • 配置显式携带 instance_id，这是正确的 namespace 来源。

需要新增的 Simulation Adapter

  • 捕获 vLLM scheduler 输出、request/block table、computed tokens。
  • 绕过真实模型权重与 attention kernel，同时保持 scheduler 可运行。
  • 把 vLLM batch 转换为统一 BatchIR。
  • 把 connector load/save 变成离散事件，不做真实大块复制。
  • 把 predictor latency 回灌到仿真时钟与请求统计。

Policy & hierarchy

7. 从“命中率”走向“有效推理收益”

缓存研究不能只报告 hit rate。一个 SSD 命中可能比重新计算更慢；一个大 prefix 的命中可能占用很多 DRAM；一个预取若错过 deadline 只会制造带宽干扰。策略目标应直接对齐 TTFT、TPOT、goodput、SLO 和数据移动成本。

先统一命中率口径

Token-weighted hit

Htoken = Σ hit tokens / Σ input tokens

最接近节省的 prefill token 数，但仍未计迁移成本。

Block-weighted hit

Hblock = Σ hit blocks / Σ requested blocks

便于研究 metadata/eviction，但最后不足一个 block 的 token 要单独处理。

Effective hit

Heff = Σ max(Trecompute − Tload − Tstall, 0) / Σ Trecompute

表示真正节省的归一化计算时间，可进一步按 SLO 权重。

策略优点缺点在本项目的状态实现建议
FIFO简单、低开销、稳定 baseline忽略访问和 prefix 结构KVCM 工厂未见维护写入序号；命中不刷新；加入策略单测。
LRU局部性强时有效；成熟 baselinescan pollution；不感知 size/load costSGLang/KVCM 均有相关路径明确 HBM radix LRU 与外部 tier LRU 是不同作用域。
LFU保留长期热点冷启动、陈旧热点、计数开销SGLang radix 实验可用，KVCM 工厂未见实现 aging/windowed LFU，避免无限累积。
RandomLRU降低精确 LRU 元数据成本方差与 sample rate 敏感KVCM 已有报告元数据/CPU overhead，而非只报告 hit。
LeafAwareLRU利用 prefix tree，减少驱逐共享祖先可能保留低价值内部节点KVCM 已有与 subtree reuse probability 联合。
TTL生命周期明确，适合 session/cache freshnessTTL 调参困难；时间语义敏感KVCM 已有滑动/固定窗口语义按 workload session gap 分布选 TTL，避免测试泄漏。
GDSF / GreedyDual-Size显式考虑 size 与 cost需要可靠 cost estimate建议 baseline把 load/recompute time 作为 cost；为多级 tier 扩展。
论文新策略联合 reuse、deadline、size、tier cost特征与预测误差可能造成反效果核心研究先用 oracle upper bound，后做轻量在线估计。

建议的新策略：Deadline-aware Reuse Value（暂名 DRV）

Value(b, i→j) = Preuse(b, Δt) · [Trecompute(b) − Tload(b, j→HBM)] − λ·Bytes(b) − μ·Interference(j)
  • Admission：只有预期净收益为正才进入 DRAM/SSD，避免 write amplification。
  • Placement：按 value density（收益/byte）选择 tier；高价值、近期 deadline 放 DRAM，低价值长尾放 SSD 或不缓存。
  • Eviction：每 tier 驱逐最低 value density；prefix ancestor 加 subtree dependent bonus。
  • Promotion：低层命中后，仅在预计再次复用且 DRAM/HBM 有预算时向上提升。
  • Prefetch：满足 predicted_ready_time ≤ compute_need_time 且不挤占更高价值 transfer。
  • 降级：预测器不确定性高时回退到 LRU/GDSF，避免 learned policy 灾难性失误。

HBM–DRAM–SSD 的状态机

转换触发是否阻塞请求需要建模
HBM → DRAMevict/write-through/background backup取决于 buffer 生命周期与 copy 完成点D2H、copy engine、pinned memory、compute overlap
DRAM → HBM命中/load/prefetch数据未在 attention 使用前到达则阻塞H2D、分层 load、batch fragmentation
DRAM → SSDdemotion/write-through通常异步，但占带宽/queuewrite QD、tail latency、write amplification
SSD → DRAMprefetch/on-demandwait_complete 阻塞；best_effort 可能部分完成read QD、page cache、并发读写争用
任意 tier → deletecapacity/TTL/invalidation通常不阻塞；metadata 有一致性窗口writing/serving/deleting 状态与失败恢复

预取控制器的四个问题

What

连续 prefix 的哪些 blocks？优先共享祖先还是当前请求独占后缀？

When

请求到达、router 预测、排队、上一个 sub-request 完成，哪个时点发起？

Where

SSD→DRAM 先热身，还是直接 SSD→HBM/GDS？目标 tier 是否有容量？

Budget

每个 copy/IO engine 的带宽、queue slots 和对关键请求的干扰预算。

Research framing

8. 论文问题、假设与实验设计

三条可答辩的研究问题

RQ1 · Fidelity

保留真实 serving control flow 的 HiSim，在跨 workload、跨 batch shape、跨硬件时，能否准确预测 TTFT/TPOT/throughput 以及策略排序？

RQ2 · Cache hierarchy

何时多级 KVCache 的 load/backup 与 contention 会抵消 prefix reuse 的收益？哪些 workload/hardware regime 决定 break-even point？

RQ3 · Policy

联合复用概率、重算成本、迁移 deadline 与容量的策略，是否比 LRU/FIFO/LFU 在不同平台和负载变化下更稳健？

建议贡献表述（必须用实验兑现）

  1. 一种 framework-in-the-loop 仿真方法：把 SGLang/vLLM 的真实调度与 cache control 归一到事件 IR，硬件执行与数据移动由可校准模型替换。
  2. 一种多级 KVCache 成本模型：同时表达 HBM/DRAM/SSD 容量、并发传输、读写争用、计算重叠和 continuous prefix semantics。
  3. 一种 deadline-aware reuse-value 策略：联合 admission、placement、eviction、promotion、prefetch；以 SLO goodput 为目标。
  4. 跨硬件/框架实证：给出误差边界、策略适用区间、反例以及可复现实验 artifact。
LLMServingSim 2.0 已宣称异构硬件扩展和约 1.9% GPU serving 误差；LMCache、Mooncake、KVDrive 等也覆盖多级缓存。因此新意不能写成“我们首次支持异构硬件/多级 KVCache”。更可信的差异是：真实框架控制流 + instance-isolated prefix semantics + 可解释的多资源竞争模型 + 策略闭环，并用明确对比证明它为何需要。
